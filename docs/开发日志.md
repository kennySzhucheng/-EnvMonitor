## 📅 2026-07-15 — 项目启动 + 方案设计

### Q: 为什么选这个方案？

- STM32F103C8T6 是最经典的学习平台，有大量社区资源，出问题容易查
- 覆盖三种总线（I2C、1-Wire、UART），学一个项目打通嵌入式通信基础
- 有显示、有输入、有声光、有无线通信——组件齐全，属于完整 IoT 终端

### Q: 为什么 OLED 和 BH1750 用同一组 I2C？

- 两者都是 I2C 器件且地址不同（OLED=0x78, BH1750=0x46），不会冲突
- 共用 PB6/PB7 省下 GPIO 给后续扩展
- 这是工程上常见做法：一条总线挂多个从设备

### Q: 为什么先面包板再 PCB？

- 面包板阶段能随便改接线，发现设计错了不受惩罚
- PCB 打样一块板子 5 元但如果改一版就是 5+5，面包板改线零成本
- 功能全部调通再画 PCB，保证一次成功

### Q: 为什么用有源蜂鸣器而不是无源蜂鸣器？

- 有源蜂鸣器通电即响，不用生成 PWM 频率信号
- 报警场景只需要"滴滴滴"，不需要播放音乐旋律
- 一个 GPIO 拉高就响，代码量和调试难度都更低

### Q: 为什么用 ESP8266 AT 指令而不是直接编程 ESP8266？

- STM32 做主控更灵活：传感器数据先在本机处理，要传什么再发
- AT 指令方式调试方便：用串口助手就能看到通信过程
- ESP8266 出厂自带 AT 固件，不用刷 SDK

### 📌 引脚分配原则

- **PA9/PA10** → USART1（ESP8266 通信），这是 STM32 固定分配
- **PA13/PA14** → 保留 SWD 调试接口（烧录程序、断点调试）
- **PB6/PB7** → I2C1（硬件 I2C），比软件模拟 I2C 更稳定
- **PA0** → DHT22 需要精确微秒时序
- **PB0/PB1** → HC-SR04 Echo 需要定时器输入捕获
- **PB12-PB15** → LED、蜂鸣器等普通 GPIO 外设

## 📅 2026-07-16（上） — 固件代码编写

### 💡 整体架构：为什么分三层？

```
Drivers/  → 硬件抽象层（"怎么读传感器"）
   ↓
App/      → 业务逻辑层（"数据怎么用"）
   ↓
Core/     → 调度层（"什么时候做什么"）
```

好处：改一个传感器不影响别的模块。比如想把 DHT22 换成 DHT11，只改 `Drivers/DHT22/` 内的代码，App 层完全不需要动。

### 💡 各驱动设计要点

**DHT22 — 为什么关中断？**
单总线时序要求微秒级精度。SysTick 中断恰好 1ms 一次，如果在读 40bit 数据时被中断打断，高电平计时就偏了，校验必错。`__disable_irq()` / `__enable_irq()` 包围整个读取过程，2ms 内完成，不影响其他任务。

**HC-SR04 — 输入捕获反转极性**
先捕获上升沿记录起始时间，立刻切到下降沿捕获结束时间。硬件定时器自动记录，不用软件轮询（会被中断干扰）。TIM3 时钟 1MHz（1us精度），`距离(cm) = 脉宽(us) / 58`。

**按键 — 状态机消抖**
每个按键 4 个状态：空闲 → 消抖等待(30ms) → 确认按下 → 长按(1s)。
30ms 过滤机械抖动，1s 触发长按，超过 1s 后每 200ms 产生 HOLD 连发事件。事件读后自动清除，不会重复触发。

**ESP8266 — 为什么用 HTTP GET 而不是 MQTT？**
Thingspeak 原生支持 HTTP GET 更新数据，不需要 MQTT broker。AT 指令模式：TCP连接 → HTTP请求 → 关闭TCP，链路清晰。后续升级 MQTT 只需改 `esp8266.c`，应用层接口不变。

**I2C 总线共用**
OLED(0x78) 和 BH1750(0x46) 地址不同，共用 PB6/PB7。HAL 的 I2C 函数是同步阻塞的，主循环里顺序调用不会冲突。上 FreeRTOS 后只需加一个互斥信号量保护。

**为什么选 Thingspeak？**
免费（每年 300 万条消息），有现成的图表和数据导出。自建服务器要花钱租云服务，不符合 100 元预算。注册：https://thingspeak.com → 创建 Channel(4个Field) → 填 API Key 到 `esp8266.h`。

## 📅 2026-07-16（下） — CubeMX 工程整合 + Keil 编译

### 💡 关键踩坑记录

#### Q1: 标准库 vs HAL 库，会不会有影响？

**不会。** 标准库学到的是寄存器思维，HAL 只是换了个函数名。初始化代码 CubeMX 生成，驱动逻辑完全自己写。标准库的 `GPIO_SetBits()` → HAL 的 `HAL_GPIO_WritePin()`，本质上操作同一个寄存器。HAL 是当前行业标准，简历和 GitHub 上更有说服力。

#### Q2: CubeMX 中 TIM3 的 Slave Mode 和 Trigger Source？

**都不选。** Slave Mode = Disabled，Trigger Source 保持灰色。HC-SR04 只需要 TIM3_CH4 做 Input Capture，不需要外部信号触发/复位/门控定时器。
- Prescaler = 72-1 = 71 → 1MHz = 1us 精度
- Period = 65535（最大计数）
- Channel4 = Input Capture direct mode

#### Q3: USART1 Mode 选什么？

**Asynchronous（异步模式）。** ESP8266 和 STM32 就是最普通的 TX+RX 两根线全双工串口，不需要同步时钟线，不需要硬件流控（RTS/CTS）。
- Baud Rate: 115200
- Word Length: 8 Bits
- Parity: None
- Stop Bits: 1

#### Q4: Clock Configuration 中 HSE 是黑的？

因为在 Pinout 页面没启用 HSE。**System Core → RCC → High Speed Clock → Crystal/Ceramic Resonator**，回到 Clock Configuration 就亮了。
- HSE(8MHz) → PLL Source Mux → PLL(×9) → SYSCLK(72MHz)
- AHB=72, APB2=72, APB1=36

#### Q5: Keil 编译 cannot open source input file "oled.h"？

头文件搜索路径没加。**Project → Options for Target → C/C++ → Include Paths → ...** → 添加 `..\Drivers\OLED` 等 9 个目录。`..\` 是从 MDK-ARM 目录向上一级到达 EnvMonitor 根目录。

#### Q6: 加了路径还是报 cannot open "oled.h"？（关键问题）

**根本原因：循环包含。**

```
main.h ──→ #include "oled.h"
               ↓
           oled.h ──→ #include "main.h"
                         ↓
                    头文件保护符(__MAIN_H)挡住
                         ↓
                    main.h 中后续的类型定义都不生效
                         ↓
                    编译器报错"找不到"
```

**解决方案：**
1. 所有驱动/应用的 `.h` 文件：`#include "main.h"` → `#include "stm32f1xx_hal.h"`
2. 所有 `.c` 文件：在 include 自己的 `.h` 之前先 `#include "main.h"`
3. `main.h` 中把所有宏定义和 typedef 移到 include 驱动头文件之前

**修复后的包含链：**
```
.c 文件 → #include "main.h" → 拿到所有类型定义
       → #include "自己.h" → 拿到函数声明
```

#### Q7: Undefined symbol SystemClock_Config？

替换 main.c 时把 CubeMX 生成的 `SystemClock_Config()` 函数体搞丢了。CubeMX 原版 main.c 底部有这个函数定义，替换后需要补回来。

## 📅 2026-07-17 — P7: FreeRTOS 任务重构

### 💡 为什么做 FreeRTOS 迁移？

裸机超级循环在简单项目是好选择，但项目有 6 个不同节奏的操作——5秒采样、1秒刷新、60秒上云、500ms报警、20ms按键、事件驱动的蜂鸣——全塞在一个 while(1) 里用时间戳区分，扩展性差。FreeRTOS 让每个功能独立运行，抢占式调度保证实时性。

### 💡 架构设计：3 层保护 + 2 条消息通道

**资源保护（互斥量）：**
- `I2C_Mutex` — OLED 和 BH1750 共享 I2C1 总线，同时访问会数据错乱
- `Data_Mutex` — g_sensorData 被 4 个任务读写，需要原子性保证
- `UART_Mutex` — ESP8266 独占 UART1，DisplayTask 手动重连和 CloudTask 自动上云可能冲突

**任务通信（队列）：**
- `ButtonQueue` — ButtonTask 扫描到事件 → DisplayTask 消费（解耦扫描和响应）
- `BuzzerQueue` — AlarmTask 发蜂鸣指令 → BuzzerTask 阻塞等待执行（蜂鸣节奏不被其他任务打断）

### 💡 6 个任务设计考量

| 任务 | 优先级 | 周期 | 关键决策 |
|------|--------|------|---------|
| SensorTask | 3 | 5s | DHT22 仍用 taskENTER_CRITICAL 关中断 ~2ms；BH1750 持 I2C 锁 |
| AlarmTask | 3 | 500ms | 高温优先级最高（continue 跳过其余）；→队列控制蜂鸣 |
| DisplayTask | 2 | 1s | 先消费按键队列再渲染；持 I2C/Data 双锁 |
| ButtonTask | 2 | 20ms | vTaskDelayUntil 精确周期；非阻塞队列发送 |
| BuzzerTask | 2 | 事件 | 阻塞在 xQueueReceive，无指令时零 CPU |
| CloudTask | 1 | 60s | 最低优先级（AT 指令阻塞 5-15s 不干扰 UI/采集） |

### 💡 SysTick 三分天下

SysTick_Handler 同时服务三个对象：
```c
HAL_IncTick();                            // HAL 库 (HAL_Delay 等)
g_sysTick++;                              // 我们自己的毫秒计数
xPortSysTickHandler();   // FreeRTOS tick (任务调度时钟)
```

关键细节：调度器启动前不调用 `xPortSysTickHandler()`，否则访问未初始化的内核结构会 HardFault。用 `xTaskGetSchedulerState() != taskSCHEDULER_NOT_STARTED` 判断。

### 💡 SVC/PendSV 为何移交给 FreeRTOS？

Cortex-M3 中：
- **SVC**（系统服务调用）：FreeRTOS 用它启动第一个任务（`vPortSVCHandler` 设置 MSP/PSP 后跳转）
- **PendSV**（可挂起系统调用）：FreeRTOS 用它做上下文切换（保存当前任务寄存器 → 恢复下一任务）

这两个中断必须由 FreeRTOS 控制。`FreeRTOSConfig.h` 中的宏 `#define vPortSVCHandler SVC_Handler` 把 FreeRTOS 函数映射到启动文件的向量表名，替代了原本的弱定义空函数。

### 💡 为什么不直接用 CubeMX 生成 FreeRTOS？

CubeMX 生成的 FreeRTOS 使用 CMSIS-RTOS v2 封装层（`osThreadNew` / `osMutexNew` 等），好处是跨 RTOS 可移植，代价是：
- 多一层封装调用开销
- 出错时调试更间接
- 学习到的 API 绑定在 CMSIS 生态

直接用 FreeRTOS 原生 API（`xTaskCreate` / `xSemaphoreCreateMutex`），社区资源丰富、面试更认可，且能深入理解 RTOS 原理。

### 💡 STM32CubeF1 自带 FreeRTOS 不用下载

STM32Cube_FW_F1_V1.8.7 固件包 `Middlewares/Third_Party/FreeRTOS/Source/` 已包含完整 FreeRTOS V10.3.1 源码。RVDS/ARM_CM3 移植层（port.c）天然兼容 Keil MDK 的 ARMCC 编译器。不用额外下载，直接从 `C:\Users\Monesy\STM32Cube\Repository\` 复制到工程即可。

### 💡 Keil 编译踩坑记录（FreeRTOS 特有）

**Q8: `#error Missing definition: configUSE_16_BIT_TICKS`**

FreeRTOS V10.x 新增的必选宏，V9.x 没有。必须设为 0（32位MCU用32位计数器）。我们漏写了，在 FreeRTOSConfig.h 补上即可。

**Q9: `configSUPPORT_STATIC_ALLOCATION` + `configSUPPORT_DYNAMIC_ALLOCATION` 必须定义**

V10.x 引入的静态/动态分配双模式。两个至少要有一个为 1（否则系统无法分配任何内存）。我们开了静态分配(=1)但没提供 `vApplicationGetIdleTaskMemory` 函数，导致链接报错。解决：关掉静态分配(=0)，只用动态分配(=1)。

**Q10: `Undefined symbol vApplicationMallocFailedHook` 等 4 个钩子函数**

FreeRTOS 的调试钩子（malloc失败、栈溢出、空闲任务内存、定时器任务内存）需要用户实现，但我们没写。这些钩子全是调试用途——关了不影响功能：
- `configUSE_MALLOC_FAILED_HOOK = 0`
- `configCHECK_FOR_STACK_OVERFLOW = 0`
- `configSUPPORT_STATIC_ALLOCATION = 0`（顺便解决了 Q9）

后续项目稳定后可以重新开启栈溢出检测（方法1/2）并补充钩子函数。

- **理念：把复杂的焊接交给工厂（核心板），把简单的插件留给手工**

### 💡 为什么用 KiCad 而不是立创EDA？

- KiCad 开源免费，无商业限制，GitHub 上更受认可
- S-expression 文件格式透明，可以用 Python 脚本辅助生成
- 立创EDA绑定嘉立创生态，后续迁移成本高
- KiCad 10.0 版本已足够成熟，社区资源丰富

### 💡 元件选择原则

核心板上已有的元件不重复画（STM32芯片、晶振x2、AMS1117-3.3V、去耦电容x9、USB、SWD、复位电路、BOOT0电路、电源LED）。PCB 上只放额外需要的外围电路。

额外增加：
- C21 (100µF/16V) +5V 电解滤波
- C22 (100µF/10V) +3.3V 电解滤波
- 所有模块接口排针排母
- 所有用户交互元件（按键、LED、蜂鸣器）

### 💡 核心踩坑：KiCad 标签不是"注释"，是"无线导线"

**坑1：标签必须放在导线上！**

在 KiCad 中，同名标签会自动合并为同一个网络——但前提是标签附着在导线上。把标签放在空白处会导致：
```
期望：R7.p2 ═══ SW2.p1 ═══ [KEY1标签] ═══ H1.p24  ← 都是同一条网
实际：R7.p2 ─ SW2.p1（net A）       [KEY1]孤悬（net B）  H1.p24（net C）
```
三条独立的网，KiCad 不知道它们要连在一起。**解决方案：把标签直接放在元件之间的导线段上，导线高亮后再放下。**

**坑2：按键消抖电路的正确拓扑**

电路要求：R（上拉）+ C（消抖）+ SW（按键）+ MCU脚 → 四者在同一节点。

错误接法（常见）：
```
+3.3V → R → [SW一脚] ─║─ [SW另一脚 + C + MCU]  ← MCU在开关后面！
```
不按时MCU脚悬空，读到随机值。

正确接法：
```
+3.3V → R → [R + C + SW一脚 + MCU]  ← 全部在同一交汇点
                    │
                  SW另一脚 → GND
```
这是经典的"上拉+消抖"按键电路，电子设计中几乎每个项目都会遇到。

**坑3：消抖电容必须接地，不是 No-Connect**

C18/C19/C20 的负极必须接 GND，不能放 No-Connect 标记。放了 × 号 ERC 不报错但电容不工作，按键会抖动。**ERC 清除 ≠ 电路正确。**

**坑4：ERC 错误数量不代表电路质量**

ERC 从 82 逐步降至 6：
- 82 → 74：修复标签悬空（5个）+ 电容接地（3个）
- 74 → 69：修复电源标签（5个）
- 69 → 6：补 C21 接线 + H1/H2 空脚加 No-Connect（~60个）+ 按键拓扑修正

剩余的 6 个是电源声明问题（`power_pin_not_driven`），因为板子由 Blue Pill 核心板供电而非自带电源芯片，这不影响功能。

### 💡 原理图检查清单方法

逐信号网络对照引脚分配表验证，是最可靠的检查方法。15 条信号线全部手动核对：
PA0→DHT22, PA4→KEY1, PA5→KEY2, PA6→KEY3, PB0→HC_TRIG, PB1→HC_ECHO,
PB6→I2C_SCL, PB7→I2C_SDA, PA9→ESP_TX, PA10→ESP_RX, PB12→LED_R,
PB13→LED_Y, PB14→LED_G, PB15→BUZZ, ESP_RST→J5.p4

全部 15 条通过后，原理图才算真正完成。

## 📅 2026-07-18 — P8: PCB 导入 + 封装修复

### 💡 原理图 → PCB 的第一步：封装必须 100% 匹配

F8 导入时 14 个元件报错（11 电阻 + 3 按键），根因是**原理图中的封装名与 KiCad 库中实际文件名不匹配**。

**坑5：Python 脚本生成的封装名不完整**

电阻的封装名 `R_Axial_DIN0207_L6.3mm_D2.5mm` 缺了脚距和方向后缀。KiCad 10 库中实际文件名为 `R_Axial_DIN0207_L6.3mm_D2.5mm_P2.54mm_Vertical`。

按键的封装名 `SW_TH_4P_5x5mm` 在 KiCad 10 标准库中根本不存在，最接近的通用封装是 `SW_PUSH_6mm`（匹配 6×6mm 轻触开关）。

**教训：** 封装名必须精确到每一个后缀字符。用 KiCad 的封装预览器（Pcbnew → 工具栏 → 浏览封装）确认名称正确后再填入原理图。

**坑6：fp-lib-table 配置**

KiCad 的封装库通过 `fp-lib-table` 文件管理。我们遇到了两个问题：
1. 手动添加库时昵称拼写错误（`Button_Switch_TH` 少了最后一个 `T`）
2. 重复添加了模板中已有的库，造成冗余

KiCad 10 自带模板库表（`template/fp-lib-table`）已包含所有标准库，一般情况下不需要手动添加。**如果添加，务必确认昵称与库文件夹名完全一致。**

**修复方法：** 直接编辑原理图 `.kicad_sch` 文件，用 sed 批量替换错误封装名，备好份后再替换。

### 💡 KiCad 学习门槛总结

今天从零开始学 KiCad 遇到的问题按频率排序：

| 频率 | 问题类型 | 典型例子 |
|------|---------|---------|
| 🔥🔥🔥 | 标签/网络概念 | 标签放空白处 = 死网络 |
| 🔥🔥 | 电路拓扑 | 按键 R+C+SW+MCU 交汇点 |
| 🔥 | 封装名不匹配 | 电阻缺脚距、按键名不存在 |
| 🔥 | 库表配置 | fp-lib-table 昵称/路径 |

**核心经验：** KiCad 的所有"自动"功能（标签合并、ERC、封装匹配）都需要输入精确——一个字符的差异就会导致失败。这和写代码一样，"语法正确"是"功能正确"的前提。

## 📅 2026-07-19（晚）— P8: PCB 布局微调 + 布线

### 💡 布局微调：消抖电容靠近按键

C18/C19/C20 是按键消抖电容（100nF），必须紧邻对应按键 SW2/SW3/SW4。移动前三组电容和按键 Y 方向差了 10mm（不同"行"），移动后 Y 差距缩小到 ~1mm，X 方向保持 11mm 左右——消抖电容离按键越近，消抖效果越好。

### 💡 电源走线：从源头出发

+5V 源自 Blue Pill H2 Pad 35（USB 5V 经核心板引出），经过 HC-SR04(J4) → 滤波电容(C21) → 蜂鸣器(LS1)。从源头出发、沿最近路径依次串联，避免星型拓扑。

由于是本项目是双面 THT 板（无贴片），所有走线都在顶层(F.Cu)，无需过孔。

### 💡 GND 铺铜：PCB 设计的标准终点

GND 不需要一根根手连——用 Copper Pour 铺铜自动连接所有 GND 焊盘。双面铺铜（F.Cu + B.Cu）提供低阻抗回流路径和 EMI 屏蔽。关键步骤：
1. 切换到目标层（B.Cu）
2. 添加 Filled Zone → 画矩形覆盖全板
3. Net 选 GND，Pad Connection 选 Solid
4. **按 B 填充所有区域**（这步最容易忘！）
5. 保存

**坑7：GND 铺铜没填充**
铺铜区域创建后只是"轮廓"，必须按 B（Fill All Zones）才能真正生成铜皮。填充前 DRC 报告 23 个 GND 相关未连接，填充后全部清除。

### 💡 布线宽度考量

全线宽 0.2mm。对最大负载（ESP8266 峰值 ~300mA），0.2mm 可承载 ~500mA（1oz 铜厚，10°C 温升）。虽然建议电源线用 0.5mm，但 0.2mm 对本板余量足够。

### 💡 DRC 验证：0 错误通过

所有 DRC 警告均为丝印重叠（R10/R11/R12 文字互碰）和封装库版本差异（H1/H2），不影响功能。**0 个未连接焊盘。**

## 📅 2026-07-20（凌晨）— 蜂鸣器方案紧急变更

### 💡 问题发现

PCB 完成后发现蜂鸣器实物是 3 脚有源模块（VCC/GND/IO，自带驱动三极管），而原理图设计了 2 脚排针 + NPN 三极管驱动电路（Q1+R13+R14+D5）。两者不兼容——3 脚模块插不到 2 脚排针上。

### 💡 为什么改原理图而不是改实物

- 改实物需要另买 2 脚蜂鸣器，等快递 2-3 天
- 改原理图只需删 4 个元件 + LS1 换 3 脚，半小时搞定
- 3 脚模块方案更简单（MCU 直连 IO），驱动电路冗余

### 💡 最小改动原则

只改蜂鸣器相关部分，不动其他电路：
- 删除：Q1（S8050）、R13（1k 基极限流）、R14（10k 基极下拉）、D5（1N4148 续流）
- 修改：LS1 从 2 脚 `Device:Buzzer` → 3 脚 `Connector:Conn_01x03_Pin`
- 连线：+5V → LS1 Pin1, GND → LS1 Pin2, BUZZ(MCU) → LS1 Pin3
- 元件从 31 减到 27

### 💡 改造流程

1. 备份原文件到 `backup_20260720/`
2. 原理图编辑器：删 Q1/R13/R14/D5 → 换 LS1 符号 → 加标签连线
3. PCB 编辑器：工具→从原理图更新 → 删残留封装 → LS1 换 1×03 → 重走 BUZZ → 铺铜
4. 验证：交叉分析 0 差异，DRC 0 错误

### 💡 KiCad 标签踩坑史

`/BUZZ` vs `BUZZ` 是两个不同的网路。第一次忘记斜杠前缀规则，告诉用户加 `/` 导致 H2.7 在 `BUZZ`、LS1 在 `/BUZZ`，网路分裂。交叉分析发现后修正为统一 `BUZZ`。

**教训：给用户指令前先确认实际网路名称，不要凭记忆猜测。**
